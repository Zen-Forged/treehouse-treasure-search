Share your booth — the Window email · v1

MVP vendor-share feature. Vendor taps the share icon on /my-shelf → bottom-sheet compose with a live preview of the 6 most-recent available finds → types one recipient email → Treehouse sends a Window email. Recipient sees a single vendor's hero name, booth post-it, mall + address, a 6-tile Window grid, and a quiet Treehouse invitation at the bottom.

What's locked (revised — second pass)

  • Q2 The Window is always 6 auto-picked tiles. 6 most-recent status='available' posts for the active booth. No manual picker in MVP.
  • Q3 One recipient at a time. Walk-up demo surface: vendor or curious shopper types one email, sends one Window. Both vendor-originated and shopper-originated shares work the same way.
  • Typography Georgia throughout the email body. Respects the committed session-32 stance in email-v1-2.html + lib/email.ts. Share email extends renderEmailShell() rather than inventing a new voice.
  • Hero weight Treehouse Finds lockup small top · vendor name is the body hero (Option 3 from the standup). Shell header at Georgia 26px/600; vendor name drops below the hairline at Georgia 34px/600.
  • Chrome Banner + post-it mirrors the /my-shelf BoothHero primitive. Hero image behind, post-it pinned bottom-right at 6° rotation. Same 196px height, same 86px post-it, same pin. Treehouse artifact recipient would recognize in-app.
  • Systems Extends lib/email.ts. New public fn sendBoothWindow(payload) + new POST /api/share-booth route. Uses renderEmailShell() for Treehouse chrome; body HTML new but composed from committed style primitives.

Build scope (pre-approval sketch)

Database

None. No schema changes for MVP. Share events logged to console.log for now; a share_events table is Sprint 6+ if analytics become a real roadmap item.

New routes and libs

  1. POST /api/share-booth — requireAuth. Reads recipient email + active vendor id from request body. Resolves the 6 most-recent available posts for that vendor via getVendorPosts(vendorId, 6). Fires sendBoothWindow(). Rate-limit 5 sends / 10 min per IP matches the /api/vendor-request pattern.
  2. lib/email.ts: sendBoothWindow(payload) — new export. Signature: { recipientEmail, vendor: Vendor, mall: Mall, posts: Post[] }. Extends renderEmailShell(); body HTML is new but composed from the shared pStyle + HAIR + PAPER primitives already declared. Preheader: "{firstName} sent you a Window into {vendorName}". Salutation-free body (no "Hi [recipient]") — recipient's name isn't known.
  3. lib/posts.ts — verify getVendorPosts(vendorId, limit) filters to status='available' and orders by created_at DESC. If it doesn't, add a new getVendorWindowPosts dedicated helper rather than mutating the existing contract. (Session-33 dependent-surface audit lesson: don't mutate a shared data function's semantics when a new caller wants different semantics — add a new function.)

Client surfaces

  1. app/my-shelf/page.tsx — share-icon button in masthead right slot. Opens a new <ShareBoothSheet> component. Entry point only shows if vendorList.length >= 1 AND the active vendor has >= 1 available post (empty-window vendors get the icon hidden — no point sharing a blank Window).
  2. New component: <ShareBoothSheet> — mirrors <BoothPickerSheet> chrome + animation. States: compose → sending → sent → error. Single email input with basic RFC validation (regex, no email existence check). CTA disabled until valid email shape. POSTs via authFetch.
  3. No BottomNav changes. Share is a vendor action, not a shopper action — no public surface affected.
  4. Q-002 coexistence. This mockup assumes Q-002 has shipped (masthead single-variant "Treehouse Finds"). If Q-002 ships after this feature, the share icon placement stays — it lives in the right slot either way.

Email rendering

  1. The post-it rotation — CSS transform is stripped by Outlook + some Gmail clients. Render the post-it as an inline SVG in the HTML email (SVG rotation is declared inside the <svg> with transform="rotate(4)" which IS respected by all modern mail clients). Mockup shows CSS rotation for review fidelity; production uses SVG.
  2. The 6 tiles — use the same Supabase public URL as the feed tiles (post-images bucket). No image resizing — modern mail clients handle the grid correctly at the source resolution. If Outlook misrenders the grid (known risk), fallback is a vertical stack of 6 images, one per row.
  3. Deep links to /vendor/[slug] — CTA button href uses the vendor's canonical public shelf URL. Already a live route; no new landing surface needed for MVP.

Rate limiting + abuse

  1. 5 sends / 10 min per IP. Matches /api/vendor-request's budget — tight enough to prevent walk-in scripting, loose enough for a vendor running a mall demo.
  2. Per-recipient dedup in-memory for 60 seconds — prevents accidental double-tap double-send if the CTA doesn't disable cleanly during sending.
  3. Abuse surface — vendor-authenticated only. Only vendors with >= 1 approved vendor row can send. Shoppers + anonymous users never see the entry point.

What QA covers at ship

  1. Vendor with empty shelf — share icon hidden ✓
  2. Vendor with 1–5 available posts — Window pads missing tiles with a graceful placeholder ("More finds coming soon" tile)? OR truncates to actual count without placeholders? Decision deferred to build-spec; mockup assumes 6 real tiles.
  3. Vendor with 6+ available posts — Window shows the 6 most-recent by created_at DESC ✓
  4. Recipient email malformed — CTA stays disabled with inline copy ✓
  5. Send succeeds — confirmation state ✓
  6. Send fails (Resend non-2xx) — error state with retry option ✓
  7. Multi-booth vendor — Window sent from the ACTIVE booth only, never a collapse of multiple booths into one ✓
  8. Vendor-uploaded image fails to load in the recipient email — alt text + rendering-degradation behavior ✓
