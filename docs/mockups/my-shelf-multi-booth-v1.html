/my-shelf multi-booth — v1 exploration

Scoping the data-model rework for vendors who operate multiple booths. Mockup shows single-booth behavior unchanged, multi-booth picker via bottom sheet (matches <MallSheet>), and two schema paths side-by-side for decision at review.

What's already decided (session 33 → 34)

  • Per-booth display_name — every booth holds its own name. A reseller with three booths has three identities.
  • Per-booth bio and hero — each booth tells its own story. No shared vendor profile.
  • Multiple pending vendor_requests per email OK as long as (mall_id, booth_number) differs. Session-32 dedup rule keys on the composite, not on email alone.
  • Bottom sheet picker — reuses <MallSheet> primitive. No new pattern invented.
  • Single-booth vendors see no picker — 99% of current users get today's UX unchanged. Picker only surfaces when vendors.length > 1.

What this mockup is asking you to judge

  • Does the masthead "active booth" affordance read as tap-able, or does it look like static text?
  • Does the bottom-sheet row hierarchy (X-glyph → mall name → booth number) feel right, or should booth lead?
  • Which schema path — A (drop UNIQUE) or B (join table) — buys the right amount of flexibility for the actual near-term roadmap?
  • Is the "Add another booth" affordance in the sheet the right placement, or should it live on a settings screen?

Schema — two paths, side by side

Both satisfy the four decisions above. Difference is migration weight and future flexibility. You pick at review.

Option A — Drop the UNIQUE

"Lighter. Faster. Right for today."

Migration

-- supabase/migrations/006_multi_booth.sql
ALTER TABLE vendors
  DROP CONSTRAINT vendors_user_id_key;

-- vendors_mall_booth_unique stays
-- vendors_slug_key stays
-- vendors.user_id remains a plain FK,
-- no longer unique

What it means

  • One auth.users row can own N vendors rows.
  • getVendorByUserId(userId) becomes getVendorsByUserId(userId) → returns an array.
  • Every current call site of getVendorByUserId updates to pick an active booth (or handle a list).
  • No new table. No new join. The data model stays flat.

Future flexibility

  • Cannot represent co-ownership (two vendors share one booth).
  • Cannot represent roles (owner / collaborator / manager).
  • If either becomes a real roadmap item, we'd migrate to Option B then.
Migration
1 SQL line
Call sites
~8 files
Session estimate
1 sprint

Option B — vendor_memberships join

"Richer. Future-proof. More surface now."

Migration

-- supabase/migrations/006_multi_booth.sql
ALTER TABLE vendors
  DROP CONSTRAINT vendors_user_id_key,
  DROP COLUMN user_id;

CREATE TABLE vendor_memberships (
  id uuid PRIMARY KEY DEFAULT gen_random_uuid(),
  user_id uuid NOT NULL REFERENCES auth.users(id),
  vendor_id uuid NOT NULL REFERENCES vendors(id),
  role text NOT NULL DEFAULT 'owner',
  created_at timestamptz DEFAULT now(),
  UNIQUE (user_id, vendor_id)
);

-- + RLS policies
-- + backfill from existing vendors.user_id
-- + indexes on both FKs

What it means

  • Membership is an explicit relationship with a role.
  • Co-ownership works: two users → same vendor, different roles.
  • Every vendor lookup gains a join. Every permission check reads role.
  • Data backfill required on migration day.

Future flexibility

  • Co-ownership: ✓ at the schema layer
  • Role-based permissions: ✓
  • Vendor-to-vendor handoffs (sell the booth): ✓
  • Business partners, hired help, guest access: ✓
Migration
~40 lines
Call sites
~15 files + RLS
Session estimate
2 sprints

Review questions

  1. Schema path — A (drop UNIQUE, lighter) or B (vendor_memberships, richer)? The roadmap currently names no near-term need for co-ownership or role-based permissions. Option A ships faster; migrating to B later is a known-weight future task.
  2. Masthead affordance — does "Viewing · ZenForged Finds ▾" read as tap-able on your phone, or does it look like static text? If static, consider: bolder chevron, underline on the name, or a subtle pill background on the whole block.
  3. Sheet row hierarchy — X-glyph → booth name (bold) → mall + booth subtitle. Does the booth name lead correctly, or should the mall lead (since that's the physical anchor)?
  4. "Add another booth" placement — inside the sheet, or on a dedicated /my-shelf/settings surface? Inside the sheet keeps the add-flow close to the list; settings keeps the sheet pure for switching.
  5. Pending-booth row — a Flow 3 request that's been submitted but not yet approved. Show in the sheet with an italic "Pending" pill? Or hide until approved?
