R3 — Admin Events tab

Supabase-backed events table capturing shopper + vendor + admin actions. New 5th tab on /admin surfaces the stream with filter chips and a 24h / 7d counter strip at top. No dashboard charts yet — raw stream first, charts only after R1 lands and the data gets richer.
Session 58 · 2026-04-24 · design record: docs/r3-analytics-design.md

Six decisions surfaced for your review — all six are reversible later

D1 — Table shape. Recommendation: single wide events table with a jsonb payload column instead of one typed table per event type. Adding a new event type becomes a one-line code change, not a migration. Aggregations still work (WHERE event_type = 'post_saved'). Typed tables create sprawl that will be regretted by R6 when the list grows.

D2 — Client vs server capture. Recommendation: hybrid. Server-side inline-writes on any event that already flows through an API route (share_sent, vendor_request_submitted, vendor_request_approved, mall_activated). Client-side fire-and-forget POST to /api/events for pure UI events (page_viewed, post_saved, filter_applied).

D3 — User identity + PII handling. Recommendation: pre-R1, user_id is nullable — populated only for vendor-authenticated events. Shopper events carry a session_id stored in sessionStorage (auto-clears on tab close), never in localStorage. No IP capture, no email capture, no user-agent capture for beta. Post-R1, shopper user_id backfills naturally when signed in.

D4 — Retention. Recommendation: indefinite retention through beta. Revisit at 90 days of data or 1M rows, whichever comes first. Beta volume makes retention cost trivially small; the compound value of historical data in the first 90 days is high.

D5 — Admin surfacing. Recommendation: stream-first, not dashboard-first. New /admin 5th tab = Events, showing 24h / 7d / all-time counter strip + filter chips by event type + most-recent-first list with 50-per-page pagination. Charts arrive in a later session once R1 makes the data richer.

D6 — v1 event list (narrow starter set). Recommendation: instrument these 8 event types now — page_viewed, post_saved, post_unsaved, filter_applied, share_sent, vendor_request_submitted, vendor_request_approved, mall_activated / mall_deactivated. Deliberately excludes post_liked (no like mechanism), share_opened (requires email redirect shim), booth_visited (folded into page_viewed via page_path).

Implementation constraints baked into the recommendation

5-tab fit on 375px width. With 5 tabs the current layout goes from "icon + label + count on one row" to "stacked icon above label" at 10px font. Shown in both frames below — tight but readable. If this crosses your bar, alternative is a separate /admin/events route.

Write volume at beta scale. Even if every shopper fires 20 events/session and beta has 50 concurrent shoppers, that's 1000 events/session — well under Supabase row-insert limits. No batching needed for v1.

Compounds with R12. Errors and analytics are the same instrumentation mindset on adjacent surfaces. R3 ships the event table; R12 (error monitoring) can reuse the same /api/events shape for structured logs if Sentry overkill for beta.
