Admin Requests tab — V1

Deny-action surface placement axis · 3 frames · realistic data

Locked Product Axes (session 131 scoping)

  • Q1 = A — Deny only (no Hold, no Needs-more-info). Pending → Approved or Denied.
  • Q2 = B — Soft email on deny; "couldn't approve, reach out if you want to discuss"; reason internal-only.
  • Q3 = B — Soft-archive (status: "denied"); chip filter surfaces past denials.
  • Q4 = A — Required reason textarea on Deny; stored in new vendor_requests.denial_reason column.

Pre-decided from Audit (no V1 axes)

  • Chip filter strip — Pending · Approved · Denied · All, default Pending (Linear/VendorsTab pattern).
  • Toast — parent-owned, success/error variants (existing AdminPage pattern, lines 975–1137).
  • Approve email — existing sendApprovalInstructions(), untouched.
  • Deny email — new sendDenialNotice(email, name), soft copy, no reason exposed.
  • R3 events — new vendor_request_denied with payload {denial_reason_length, mall_id, booth_number} (length not text — internal audit only).

The One Open Axis This V1 Spans

  • Where does the Deny + required-reason textarea live?
  • Frame A — Inline two-button row + confirm modal with textarea (Substack/GitHub-inspired).
  • Frame B — Expand-on-tap accordion + inline textarea (Notion-inspired; mirrors VendorsTab D7).
  • Frame C — Single Review CTA → modal-driven decision with proof image + textarea (Stripe-inspired).

Frame A

Inline buttons. Confirm modal. Co-equal Approve/Deny.

Each row carries two pill buttons right-aligned. Approve = 1 tap (existing). Deny = tap → confirm modal with required textarea + soft-email preview → confirm. Reason textarea lives in modal only. Denied rows still appear (when Denied chip on).
