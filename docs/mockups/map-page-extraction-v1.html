Trade-off matrix

Axis Frame A (bordered) Frame B (soft-padded) Frame C (full-bleed)
"Contained window" feel Strongest Moderate Weakest
"Better use of screen" Smallest map area Mid map area Largest map area
Chrome separation from map Clear (border + bg) Clear (shadow + bg) Soft (gradient overlay)
Map-edge label readability Always visible Always visible Top labels under masthead-fade
"Seamless from drawer" feel Reads as discrete page Mid (familiar chrome) Map IS the surface (closest to drawer expansion)
Implementation complexity Lowest (standard chrome stack) Lowest Higher (overlay + gradient + z-index)
Substrate reuse 100% (existing components) 100% 100% (chrome positions differ only)
David's verbatim ask: "I want to pull the map portion back out from the explore page into it's own page. My goal is to still make it feel like a seamless transition though as much as possible. It needs to have a back button to go back to the explore page. It should not include a search bar. I want it to make better use of the screen and still feel part of the app. I liked the first iteration where it felt a bit contained in a window an not extending the entire screen."

Tension to resolve: "better use of screen" pulls toward C; "contained in a window not extending entire screen" pulls toward A. Frame B is the middle-ground hypothesis that honors both. Frame C maximizes the "seamless from drawer" feel (drawer was already a chrome-less expanding overlay).
