Rich PostcardMallCard — V1

Three structural frames spanning the photo + search-bar layout axis. V1.1 refresh — photo gets rounded-corner treatment inset from card edges, mall name sized to fit one row at our card width, "select location" rendered in Dancing Script.

What V1 is asking you to judge

Pick the structural shape for the rich card: where does the photo go, where does the search bar go, how do those two affordances co-exist with the existing eyebrow + name + address + stamp vocabulary?

Locked from prior conversation

  • Surface scope: rich card on Home only. Map + Saved keep current slim card unchanged.
  • Empty-state collapse: when hero_image_url is null OR scope is all-Kentucky, drop photo region but keep search bar inside the card.
  • Address format: full street + city + state + zip (today's mall.address) — important as locations scale.
  • Eyebrow "Discover finds from:": existing FONT_LORA italic.
  • Wiring: slim card stays in (tabs)/layout.tsx for Map+Saved; rich card mounts inline on Home only (option b2). FeaturedBanner mall-hero mount + standalone SearchBar mount come out of app/(tabs)/page.tsx.
V1.1 deltas from first draft:
  • Photo inset 14px from card sides + bottom, 8px corner radius — reads as a card-within-a-card, not a banner that bleeds to the edge.
  • Mall name dropped to 22px (production: fluid 22→16 measure-and-shrink). "Copper Awning Flea Market" now fits one row.
  • "select location" rendered in Dancing Script (was Caveat in first draft). New addition to the font library; used sparingly + intentionally for personal-touch labels only.
  • Stamp-area auto-widths so the script label can't overflow off the card.

Questions for V1 → V2

  1. Frame pick — A, B, or C. ("A but with X from B/C" is fine too.)
  2. Script label — does Dancing Script land for "select location"? If it reads too curly / too wedding-invitation, alternatives in V2 could be Caveat (more handwritten-pen), Kalam (more casual), or Sacramento (between the two). Or stay with Dancing Script and refine size/rotation in V2.
  3. Frame C's empty state shape difference — when no photo, C's search bar can't inset onto a non-existent photo, so it falls back to a normal pill row below the stamp. OK, or does this disqualify C?
