What stays in every state

  • Wordmark masthead, search bar, mall scope block, hairline, BottomNav. No surfaces appear or disappear based on search state — only the masonry below the hairline changes.
  • Mall picker stays — search respects the current mall scope by default. The "Search all of Kentucky" link only appears in the empty state, as a contextual escape hatch.
  • The filter glyph (right side of bar) is inert in phase 1. Phase 2 hook for axis-narrowing (material / era / object) if shopper data shows demand.

What changes between states

  • Idle → Active: bar gains a 3px green ring + lifted shadow + opaque background. Clear (×) glyph appears next to the typed query.
  • Active with results: masonry re-sorts to matching tiles. No extra labels, no "12 results" header, no matching-reason chips. Per the design philosophy: "we don't need to call it out anywhere."
  • Active empty: masonry replaced by a single italic Lora line + outlined green CTA. Both phrased contextually: "Nothing matching 'X' at this location yet" + "Search all of Kentucky →".

What ships in phase 1 (UI)

  • One <SearchBar> primitive, slotted between StickyMasthead and the mall picker on Home.
  • Glass-morphism CSS exactly as David specified. backdrop-filter: blur(10px) works in iOS Safari 9+ and is harmless on browsers that don't support it (falls back to the 65% white background).
  • Debounced 200ms, query writes to ?q=brass. URL-shareable. Browser back exits the search cleanly.
  • Empty-state escape hatch routes to the same Home with mall scope cleared (?q=brass&mall=all).

What ships in phase 1 (data layer — the actual work)

  • New migration 019_post_tags_and_search.sql: adds posts.tags text[] + a generated tsvector column over title || caption || tags || vendor.display_name || mall.name + a GIN index on it.
  • Enrich existing /api/post-caption (Sonnet 4.6 vision call already firing on every find) to also return 5–6 categorical tags in the same JSON response — same image, same model, ~+150 tokens output. No new AI route, no extra round-trip per post.
  • One-shot backfill script scripts/backfill-tags.ts runs /api/post-caption-style enrichment over existing rows. Cost estimate at ~120 posts: ~$0.50 in Sonnet vision calls.
  • Search query: single Postgres call with websearch_to_tsquery('english', $1) against the GIN index. Sub-50ms even on 10K rows.

Out of scope (phase 1)

  • Filter glyph is non-interactive. No axis chips, no sort, no "by material" UI.
  • No search history, no recent-searches dropdown, no autocomplete.
  • No matching-reason metadata on tiles ("matched: tag" chips). The magic is that it just works.
  • No analytics on what shoppers search for — comes free via R3 if we add a single search_performed event later (~5 min).

Open implementation questions

  • Should the search bar persist when scrolling (sticky below the wordmark) or scroll away with the masthead? Default: scrolls away — sticky search bar is heavier visual chrome and the user can swipe up to expose it.
  • What's the tag taxonomy we ask Claude for? Suggested 6 axes: material, era, object_type, color, subject, category. Reusable from /api/identify's schema (the reseller layer's prompt) but trimmed.
  • Empty-state copy — keep "Nothing matching 'X' at this location yet" or shorten to "No matches yet"? Long form gives more "Treehouse voice"; short form is faster to read.
