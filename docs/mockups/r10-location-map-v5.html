R10 — /map page body · v5

Postcard card + 4-tab nav locked in v4. v5 designs the map body itself: basemap visual treatment, pin-tap interaction, search behavior on the map page, and what each scope state looks like.

v5 mockup · 2026-05-04 · 3 page states + basemap + interaction picks

Three open questions for v5

  • Basemap visual style — cartographic warm-cream (V1 Frame A direction; matches v1.x ecosystem palette) vs. clean light Mapbox/Google standard (familiar but generic). Two stages below.
  • Pin-tap interaction — instant rescope per design-record D23 (one tap = postcard card updates + map zooms in) vs. peek-then-commit (first tap = inline callout near the pin; tapping the callout commits the rescope). Two stages below.
  • Search bar behavior on Map — not covered in the design record. Three options surfaced below: filter-pins / redirect-to-Home / disabled-on-Map.

Three page states (frames below)

  • Frame A — All Kentucky overview. Default state when user first taps Map tab. Card shows "All Kentucky Locations." Map shows all active mall pins on a state-shaped basemap. No selection.
  • Frame B — Specific mall scope. User has set scope to one mall (via MallSheet from any tab). Card shows that mall. Map zooms to its location with a single highlighted pin + neighborhood context.
  • Frame C — Pin tapped, peek callout. All-Kentucky scope with one pin tapped. Inline callout appears above the pin showing mall name + booth/find counts + chevron. Tap the callout → commits the rescope (Frame B). Tap empty map → dismisses.

Recommendation: warm-cream cartographic basemap + peek-then-commit interaction + redirect-to-Home for search-on-map. Reasoning in each pane below.

What's NOT being designed in v5

  • Map provider implementation. Mapbox / Google Maps / Leaflet decision still deferred to the implementation arc.
  • Pin clustering. 5 active malls in Kentucky; clustering only matters at 15+ locations.
  • "You are here" GPS indicator. Permission flow + privacy-pref handling are post-launch polish unless beta feedback demands earlier.
  • Bounding-box constraint logic. Cap pan/zoom to KY at implementation time.

Basemap visual style — pick one The cartographic warm-cream is recommended (matches v1.x palette). Standard light is the fallback if cartographic styling is too costly to implement on the chosen provider.

Cartographic warm-cream · recommended
Custom Mapbox style. Cream landmass, sage water, white roads, italic Lora region labels.
Pros: Strongest "doesn't feel tacked on" read. Composes with paperCream / Lora / leaf brand mark. Italic region labels echo the in-app cartographic eyebrow on /find/[id].
Cons: Custom styling effort. Mapbox Studio styling pass needed (~half-day). Tile cost slightly higher than standard styles but well within Mapbox's free tier at projected traffic.
Standard light
Mapbox-light or Google Maps default. Familiar to all shoppers; renders without custom styling.
Pros: Zero custom-styling cost. Familiar visual language for all shoppers. Faster ship.
Cons: Reads as Airbnb/Yelp-generic. The leaf pins do most of the brand work; basemap stays neutral. Risks "tacked on" feel against the otherwise-handcrafted v1.x ecosystem.

Pin-tap interaction — pick one Refines design-record D23 (which currently specifies instant rescope). Peek-then-commit is a cheap UX upgrade for users who want to compare before committing.

Instant rescope (per current D23)
One tap on a pin = postcard card animates to that mall + map zooms in.
Pros: Fewer steps. Card top is the only mall identifier needed. Composes cleanly with the no-bottom-card thesis (D22).
Cons: User has to commit before seeing what's at a mall. Friction for "compare before deciding" mental model. Smaller relative friction at 5 malls; scales worse if KY grows to 15+.
Peek then commit · recommended refinement
First tap = small inline callout near pin. Second tap (callout) = commit. Tap empty map = dismiss.

Cons: Two-tap commit is one extra step. If only 5 malls existed forever this is overkill; for growth-state UX it's worth it.

Search bar behavior on the Map page The search bar primitive ships to all 3 tabs (D20). On Map, what does it do?

Three options

(a) Filter pins in place

Type "brass" → pins of malls with no brass finds dim or hide. Map stays. Subtle, immersive.

Cost: requires an aggregation query (mall.id × tag-match count). Not in current schema; would need a derived index or runtime query per keystroke.

(b) Redirect to Home · recommended

Type "brass" + submit → routes to /?q=brass. Search lives on Home; Map stays focused on geographic browse.

Cost: trivial. Reuses existing R16 search on Home. Mall scope persists across the redirect via URL.

(c) Disabled placeholder

Search bar present for visual consistency, but interactions disabled. Tap shows toast: "Search lives on Home."

Cost: trivial. But dishonest UI: present without function.

Recommendation: (b) Redirect to Home. Cleanest mental model — Search is a Home action, Map is a geographic action. Redirect preserves mall scope; user lands on Home with the same mall filtering plus the typed query active. Zero new infrastructure. Doesn't violate Map's "this is the geographic surface" focus.
