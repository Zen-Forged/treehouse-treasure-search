Admin Vendors tab — V1

Open structural axis: row shape + how affordances surface. Status, filter chips, and proactive collision detection are constant across all three.

Frozen by analysis (constant across V1 frames)

  • New tab as slot 6 — Requests · Posts · Banners · Malls · Events · Vendors.
  • Default scope: Problematic only — chip toggle to All. Problematic = unlinked (orphan) + collision (display-name diff vs approved request).
  • Status badges per row — Linked (green) · Unlinked (amber) · Collision (red).
  • Affordances available on every row — Edit (display_name, booth#, mall) · Force-unlink (with "X loses /my-shelf access" warn) · Force-delete (warn-then-allow with N-posts cascade) · Relink (assign to different vendor_request → fixes display_name + sets user_id).
  • Proactive collision detection — fanout /api/admin/diagnose-request on tab load for every unlinked row; collision badge surfaces inline, not on-demand.
  • Add Booth folds here — retires from Requests tab; surfaced as a + CTA on this tab.

Open in V1 (the structural axes this mockup explores)

  • Frame A — Compact list, overflow menu. Density-first; Linear/Vercel pattern. Tap ⋯ → popover (Edit / Unlink / Delete / Relink). Edit opens existing booth-edit-sheet.
  • Frame B — Expandable accordion. Context-first; Stripe pattern. Tap row → reveal full metadata + 4 inline action buttons. No sheet for inspection.
  • Frame C — Card-per-vendor, all actions visible. Discoverability-first. Heaviest chrome, slower scroll, but unfamiliar admin learns by seeing.

Questions for David

  1. Pick a row shape. Frame A — compact list w/ overflow menu, Frame B — expandable accordion, or Frame C — card-per-vendor? My lean: Frame B. Vendors-tab usage is investigative, not bulk-actiony — when you open it you usually want the full picture of one or two specific rows (the booth-456 case from session 123 is the canonical example). Accordion gives you that without a sheet hop, while keeping the list scan-friendly when collapsed. Frame A is great for daily ops with high-volume changes, but you'll spend more time investigating individual stale rows than batch-managing the whole table.
  2. Edit affordance — sheet or inline? Frame A implies "Edit opens existing booth-edit-sheet" (consistent with session 74). Frames B and C could either open the sheet OR convert their detail-row into editable fields. My lean: open the existing booth-edit-sheet for all frames — reuses session-74 work, no new edit primitive, smaller blast radius. Counterargument: editing inside the row is more "Stripe-dashboard-modern" if you pick Frame B.
  3. "Add booth" placement. Mockup shows a dashed-pill CTA top-right of the body. Alternatives: floating action button bottom-right, dedicated "+ row" at top of list, or fold AddBoothInline as a permanent expanded row. Lean: top-right dashed-pill as drawn — minimal chrome, retires AddBoothInline cleanly.
  4. Collision row visual treatment. All three frames use a red left edge + tinted row background + red collision pill. Too loud? Subtler treatment? My lean: keep as drawn — collision rows are rare (1-2 per session) and you'll want them screaming for attention.
  5. Filter strip vs segmented control. Mockup uses pill chips (Problematic / All + Collision / Unlinked). Could be a segmented control instead. Lean: chips as drawn — matches Saved page filter chip vocabulary, multi-select capable if we want stacked filters later.
  6. Anything missing. What admin task do you do today via SQL paste that this design wouldn't let you do? Stale-row scan, bulk-delete by name pattern, vendor → posts deep-link, audit trail (who approved, when), anything else?
